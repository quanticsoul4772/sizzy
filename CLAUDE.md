# CLAUDE.md

Entry point for Claude Code sessions in the devharness repo.

## What this project is

devharness is a phased, event-sourced agent harness for collaborative software development.
Four roles with structurally-enforced tool boundaries — research, director, developer,
reviewer — drive a research → spec → plan → write → verify → review → integrate loop, under
operator-in-the-loop gates. Load-bearing properties: verifier-first acceptance (a task is
`completed` only when a declared verifier passes AND a fresh-context reviewer certifies —
Invariant 5), a single-writer lock, fail-closed gates, an OSS-contribution envelope, and a
learning spine whose proposals never auto-apply. The planned B0–B5 rollout is complete and
CI-green; all 18 invariants are enforced with tests. The design of record is `devharness-spec.md`
(and its revision history); a compressed feature history is in `CHANGELOG.md`. (This file's
session-by-session build narrative is kept in the private archive, not the public mirror.)

## Architecture at a glance

The shape an agent needs before opening the spec. Source of truth is `devharness-spec.md` §Architecture and §Invariants.

**Four roles, one writer.** A role's identity *is* its enforced tool boundary, not a prompt persona:

| Role | Can | Cannot | Invariant |
|---|---|---|---|
| Research | read/research tools, fan-out sub-agents | repo writes | — |
| Director | dispatch, plan, mcp-reasoning | any file write | 3 |
| Developer | hold the single write lock; edit/commit in one isolated worktree | write outside its task scope | 1 |
| Reviewer | read, parallax `verify`/`check`, run tests | any write/edit/commit | 2 |

**The loop:** research (with operator) → spec → **operator sign-off gate** → director plans → developer writes one task in isolation → reviewer certifies → director integrates → ship → maintain. Only the developer touches code. Human gates: spec sign-off, and (early) integration.

**`completed` is earned twice** (Invariant 5): declared verification passes *and* the reviewer certifies in a fresh context — separate checks, both required. A developer asserting "done" does nothing.

**Cost tiers T0–T3** (deterministic → frontier): advisory roles bias T0–T1; the single writer runs T2–T3; the director's tier minimum is declared per task class (§S2).

**The 18 invariants are the test contract.** Each MUST have a test (spec §Invariants). Any design change preserves all 18 or amends the spec first.

## Conventions for working in this repo

This is a fresh project. There is no inherited operator-discipline hook layer. The conventions below come from the spec itself; they bind anyone authoring devharness code, including agents.

- **Spec is the source of truth.** Changes to design land as spec edits first, with a revision-history entry. Code follows the spec, never the other way around.
- **Constitution amendments carry a semantic version bump** (Invariant 18). Edits to `.specify/memory/constitution.md` change the `Version:` line and add a row to the amendment log at the bottom of that file.
- **Forward-only schema migrations**, numbered under `schema/migrations/` (`0001`–`0028` so far; the runner fails closed on a gap or out-of-order). Projection surrogate keys avoid `AUTOINCREMENT` so a DELETE+replay rebuild reproduces rowids (Invariant 8 parity, spec §Data model); empty placeholder tables may be DROP+CREATEd, data-holding tables need ALTER/CTAS. Each migration breaks the three ledger-idempotency tests (`test_artifacts_migration`, `test_b1_projections_migration`, `test_lock_migration`) until the new version is appended to their `applied_versions` assertion.
- **The event log is the telemetry** (Governing Layer commitment 5). No additive observability layer (OpenTelemetry, Langfuse, etc.). If an observation matters, it is an event.
- **`EventBus.emit_sync()` is the sole writer to the event store** once the runtime exists. Direct SQL writes to the events table are a structural violation.
- **`setting_sources=[]` posture** for any Claude Agent SDK invocation (Governing Layer commitment 3). No silent inheritance of CLAUDE.md or settings into agent sessions.
- **Single-writer lock** governs developer-role code-mutating sessions (Invariant 1). Worktrees are serial under the lock; never two writers at once.
- **Verifier-first acceptance plus reviewer certification** (Invariant 5). A `completed` terminal requires both, separately.
- **Antibodies are text only** (Invariant 11). Retro proposals that contain code go to the gate-change queue, not the antibody queue.
- **Core gates cannot be weakened by retro proposals** (Invariant 12). Rejected at the validator.
- **Developer writes are scope-enforced on the realized worktree diff** (rev 0.3.21), not only on ACI editor tool-calls — `DeveloperRole._enforce_worktree_scope` rejects + rewinds any out-of-scope change regardless of write vector (editor / shell / built-ins). Built-in `Write`/`Edit`/`Bash` are disallowed (rev 0.3.22) so the only write path is the ACI; the ACI shell/test-runner route through a §S5 `SandboxLauncher` when `DeveloperRole(sandbox_launcher=…)` is set (rev 0.3.24, opt-in).
- **Research synthesizes the spec body; the director decomposes the spec** (rev 0.3.23) — `roles/synthesis.py` (`parse_spec_body`/`parse_task_list`) strictly validates the model's JSON and **falls back** to the prior templated body / single-task default on malformed output, so synthesis only ever upgrades the result. The free-form `MCPClient.complete()` is the synthesis/decomposition query path.
- **Test-affordances are explicit and default-off.** Some production classes carry a test-only seam so a test can inject behavior the mocked path won't produce; these are back-compatible (no-op / unused in production). Current ones: `DeveloperRole(write_hook=...)` (synthetic worker writes under the lock, since the SDK worker is mocked in tests — `runtime/devharness/roles/developer.py`), `clear_task_classes()` (registry reset — `runtime/devharness/task_classes/registry.py`), and `ConsoleDeveloper(retro_engine=...)` (injects the §S7 residue engine so the post-build auto-drain runs T0-only/faked instead of building the live T1 parallax client — `runtime/devharness/console/developer.py`). Keep this list current when adding more.
- **Dev-stack teardown must not blanket-kill node (🔴 operational invariant).** When standing up the local dashboard for a browser re-verify (sidecar + vite), tear it down with `taskkill //IM sidecar.exe //F` plus the **vite PID by port** (`netstat -ano | grep ":5173" | grep LISTENING` → `taskkill //PID <pid> //F`). **Never `taskkill //IM node.exe //F`** — the Playwright MCP server itself runs as a `node.exe` process, so a blanket node kill drops the MCP. This was the root cause of the Playwright re-verify deferring across B1.7/B2.9/B2.10/B3.8/B3.9 (each teardown killed the MCP); identified and fixed at B3 closure. The B4.7 28-tile re-verify ran cleanly under this pattern (MCP preserved).
- **Disable `core.fsmonitor` when driving the loop (🔴 operational invariant).** Git for Windows enables `core.fsmonitor=true` in its *system* config, so every git op spawns a detached `git fsmonitor--daemon`. The harness creates and destroys a **worktree per task** and fires many git ops per worktree, so git's daemon-reuse singleton never settles — each op orphans a fresh daemon that never exits. A multi-task drive (e.g. jqlite's 9 features + bugfix) accumulated **1,439** orphaned daemons, whose handle/process-table pressure pushed `claude.exe`/MCP spawns past the SDK's **60s `initialize` timeout** → intermittent `Control request timeout: initialize` worker/verifier failures (a plain re-run recovered each, because it just needed an uncontended moment). Fix: `git config --global core.fsmonitor false` (overrides the system default; repo-local set too) — already applied on this machine. If you see those init-timeouts pile up, check `wmic process where "name='git.exe'" get commandline` for `fsmonitor--daemon` and `taskkill //F //IM git.exe` to clear the zombies (safe — they're detached daemons; no real git runs between tasks).
