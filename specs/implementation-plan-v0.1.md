## 1. B0 goal and non-goals

**Goal.** A booting three-process substrate where: every state change is an append-only, hash-chained event written through a single writer; projections rebuild from the log with proven parity; the dashboard shows live tiles fed by an SSE sidecar; and boot fails closed unless all thirteen commitments map 1:N to a registered boot-check set (Invariant 18) and all REQUIRED_GATES are present (Invariant 15).

**Non-goals in B0** (deferred to the named cut line):
- No research / director / developer / reviewer behavior (B1+).

- No task classes, gates beyond boot checks, calibration, cost model, OSS envelope, learning loop. Those land B1–B5.
- No CLAUDE.md/settings inheritance plumbing for agents — `setting_sources=[]` posture is asserted by a boot check now, exercised when agents arrive.

## 3. Three-process topology

One SQLite file (`*.db`, gitignored) is the shared substrate; WAL mode permits one writer + concurrent readers (spec §Assumptions: SQLite + WAL + FTS5 + sqlite-vec).

| Process | Lang | Role in B0 | Reads | Writes |
|---|---|---|---|---|
| **runtime** | Python | Owns the event store and projections; runs boot checks fail-closed at startup; the only writer. | event log, projections | event log via `EventBus.emit_sync()` **only** |
| **sidecar** | Rust | Tails the event store and fans out per-channel SSE streams to the dashboard. Read-only on the DB. | event log (read) | nothing (SSE out) |
| **dashboard** | Svelte | Renders 12 live tiles from SSE; polling cadence slows when paused (spec §S9). | SSE from sidecar | nothing |

**Single-writer invariant at the substrate level (Invariant 1 / commitment 11):** `EventBus.emit_sync()` is the sole code path that writes the events table. Direct SQL writes are a structural violation. The sidecar and dashboard are read-only by construction (the sidecar opens the DB read-only; the dashboard never touches the DB). This is distinct from the *developer write lock* (also Invariant 1) which governs code-mutating agent sessions and is not exercised until B2 — B0 only ships the lock primitive and its boot check.

**Cross-process independence (L8):** every "X survives when Y fails" claim is backed by separate processes and an explicit degraded-state test (e.g. dashboard shows a stale/disconnected banner when the sidecar dies; runtime serves and writes when the dashboard is down). No single-process shortcut that would make the independence claim untestable.

## 4. Schema and migrations

- **Forward-only, numbered** SQL files in `schema/migrations/`: `0001_initial.sql`, `0002_*.sql`, … Never edit a shipped migration; never renumber; never down-migrate. A correction is a new higher-numbered migration. (Mirrors the pgharness pattern; CLAUDE.md "Conventions".)
- A migration runner in `runtime/` applies pending migrations in order at boot and records applied versions in a `schema_migrations` table. Applying out of order, or a gap, fails closed.
- **`0001_initial.sql` ships the B0 substrate tables:**
  - `events` — append-only log. Columns include monotonic `seq`, `event_id`, `correlation_id` (NOT NULL — Invariant 9), `event_type`, `payload` (JSON), `prev_hash`, `hash`. The hash chains over the prior row's `hash` + this row's canonical content (Invariant 7).
  - `schema_migrations` — applied version ledger.
  - Projection tables (read models the dashboard needs; see §6), all derivable by replay.
  - FTS5 + sqlite-vec scaffolding stubbed for B5 memory; not populated in B0.
- **Boot integrity check (Invariant 7):** on startup the runtime recomputes the hash chain over `events`; any break (manual mutation, gap) fails boot with a deny envelope.

### 5.2 The parity gate (Invariant 18)

A CI test (`tests/runtime/test_constitution_parity.py`) that:
1. **Parses** the constitution at `.specify/memory/constitution.md`, extracting each commitment's declared claim set from its `Boot-check claim set:` block. Result: a `{commitment_id → set(names)}` map; flattened, the 19 names.
2. **Loads** the runtime boot-check registry (the registered `check_*` callables, each tagged with the commitment it serves).
3. **Asserts both directions:**
   - every declared name is present in the registry (no missing check), and
   - every registered check maps back to exactly one commitment (no orphan check).
4. Fails CI on either an unmapped commitment name or an orphan boot check.

The same assertion runs at boot (fail-closed) so a registry/constitution drift cannot start the runtime, not just cannot pass CI. Per constitution §Amendment process and Invariant 18, any change to a claim set must land with a matching registry change in the same commit, and a constitution claim-set change is a semver bump (minor for a new claim/commitment).

### 5.3 B0 boot-check semantics

B0 makes each of the 19 checks *real but scoped to substrate truth* — none is a stub returning `true`. Where the thing it guards is not yet exercised until a later cut line, the check verifies the *primitive's presence and shape*, not agent behavior. Examples:
- `check_event_log_writer_singleton` — asserts `EventBus.emit_sync()` is the only registered writer path (static + runtime guard).
- `check_correlation_id_coverage` — asserts the events schema makes `correlation_id` NOT NULL and the writer refuses an event without one.
- `check_projection_rebuild_parity` — asserts a full replay reproduces incremental projection state (the §7 parity test, run as a boot check).
- `check_single_writer_lock_present` / `check_concurrent_write_attempts_fail_closed` — assert the lock primitive exists and a second acquire fails closed (no agent needed to test the primitive).
- `check_setting_sources_empty` — asserts the Agent SDK invocation config carries `setting_sources=[]` (the config exists in B0 even though no agent runs).
- `check_dashboard_tile_coverage` — asserts the 12 tiles (§6) are all declared and wired to a channel.
- `check_required_gates_registered` — asserts REQUIRED_GATES (Invariant 15) is non-empty and all members present. Per spec §S5 the four OSS fear-map gates (workflow_guard, secret_guard, scope_guard, sandbox) are *in REQUIRED_GATES from boot*, so B0 registers all four as fail-closed, deny-by-default gates: each denies any OSS (`is_oss=True`) action, and no such task exists until B4. B0's REQUIRED_GATES = the four fear-map gates + the spec-gate primitive + the boot-check set. B4 fills in the gates' allow-logic and the sandbox/intake/caps machinery; it does *not* add them to REQUIRED_GATES (they were already required). This keeps Invariant 15 literally true from the first boot rather than parameterizing the required set by cut line. (An earlier draft excluded the OSS gates from B0; parallax `verify` refuted that as contradicting Invariant 15's literal text — corrected here.)

Any check that cannot yet be made meaningful at substrate level is a signal the commitment isn't a B0 concern — it is flagged in this plan's §10 for the operator rather than shipped as a fake green.

## 7. Projections and parity

- Projections are derived read models (the §6 tiles' backing tables). They are rebuildable: dropping and replaying the full event log reproduces them exactly.
- **Parity test (Invariant 8 / `check_projection_rebuild_parity`):** maintain incremental projection state as events append; periodically (and in CI, and as a boot check) rebuild from scratch and assert byte/row equality with the incremental state. Divergence fails.
- This is the substrate guarantee the whole audit/telemetry/replay story rests on (spec §Spine).

## 8. Adopt-from-birth lessons L7–L11

From `developer-agent-harness-brief-revised.md` §"Adopt-from-birth lessons (v3's L7–L11)". Baked into B0, not retrofitted:
